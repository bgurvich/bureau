# Secretaire тАФ Roadmap

Personal-affairs management app. **Primary purpose: grasp the big picture of life state** тАФ synthesis, dashboards, timelines. Detail capture is a means, not the end.

Stack: Laravel 13 ┬╖ Livewire 4 ┬╖ Vite 8 ┬╖ MariaDB ┬╖ stancl/tenancy (single-DB v1 тЖТ DB-per-tenant at SaaS).

Status legend: тЬЕ in v1 schema ┬╖ ЁЯПЧ v1 wiring (post-schema) ┬╖ ЁЯзн deferred, schema-reserved ┬╖ ЁЯТб brainstormed, no schema yet.

### Fiscal (hybrid accounting)
- тЬЕ Accounts (bank / credit / cash / investment / loan / mortgage)
- тЬЕ loan_terms (1:1 side table for loan/mortgage accounts тАФ principal, rate, maturity)
- тЬЕ account_balances (daily snapshot rows тЖТ net-worth line)
- тЬЕ Categories (expense / income / transfer kind)
- тЬЕ Transactions (signed amount, currency, category, counterparty)
- тЬЕ Transfers (dedicated table тАФ from/to account, from/to amount, from/to currency)
- тЬЕ recurring_rules (RFC-5545 RRULE, unifies bills + income + transfers + maintenance + warranties)
- тЬЕ recurring_projections (materialized upcoming occurrences тЖТ time radar)
- ЁЯПЧ **Bills as first-class surface (unified recurring + one-off)** тАФ a "Bill" is a `recurring_rule` plus its next `recurring_projection`, no new table. Schema + autopay UX shipped; paper-bill intake + auto-matcher + reconciliation workbench still pending.
  - **DTSTART = issue date** (invoice date, when the bill was created by the counterparty), **not** the due date. The two are distinct and both matter (issue = accrual, due = cash-flow).
  - Recurring bill: RRULE anchored on the typical issue day (`FREQ=MONTHLY;BYMONTHDAY=1` тАФ landlord issues on the 1st). Each occurrence has a due date computed as `issued_on + due_offset_days`.
  - One-off bill (medical, repair, surprise invoice): RRULE `FREQ=DAILY;COUNT=1`, DTSTART = the actual issue date on the paper. Expander emits exactly one projection on the issue date; its `due_on` is issue + `due_offset_days`. Rule effectively inert after its single projection matches a transaction.
  - **Schema additions** (тЬЕ shipped via `2026_04_17_116000_add_bill_fields_to_recurring_tables.php`):
    - тЬЕ `recurring_rules.due_offset_days` (unsigned small int, default 0) тАФ days between issue and due for occurrences generated by this rule.
    - тЬЕ `recurring_rules.autopay` (bool, default false) тАФ bill is on the counterparty's auto-charge; no manual payment required. Propagates to projections.
    - тЬЕ `recurring_projections.issued_on` (nullable date) тАФ the expanded DTSTART date for that occurrence (= RRULE anchor). `due_on` stays as the payable date.
    - тЬЕ `recurring_projections.autopay` (bool, default false) тАФ denormalized from the rule at projection time so later edits to the rule don't rewrite history.
    - тЬЕ The `recurring:project` command fills `issued_on` with the expander's date, `due_on = issued_on + rule.due_offset_days`, and copies `autopay` from the rule.
  - **Autopay behavior** (тЬЕ shipped):
    - тЬЕ Attention radar skips autopay projections until they're more than 7 days past `due_on` without a matched transaction тАФ at which point something failed (expired card, declined charge) and the user genuinely needs to look.
    - тЬЕ Bills & Income page still renders autopay projections but in a muted "informational" row style with an `AUTO` chip, so the user sees upcoming burn without being nudged.
    - ЁЯзн Money radar's "next 30 days out" cumulative burn includes autopay projections (they still hit the account).
    - ЁЯзн When bank feeds + reconciliation land, autopay projections auto-promote to `matched` on any incoming bank row that satisfies the match rule.
  - тЬЕ **Paper-bill intake workflow**:
    1. тЬЕ Bill arrives (mail, email, or verbal). User hits `тМШ.` тЖТ **Bill** in the Inspector type picker, or `B` keyboard shortcut on `/bills`.
    2. тЬЕ Form fields shipped: `counterparty` (searchable-select with allow-create), `amount`, `issued_on` (defaults today), `due_on` (required; inspector computes `due_offset_days` from the two dates), `category`, `account`, `is_recurring` toggle тЖТ monthly/weekly/yearly frequency + optional stop-after; off = `COUNT=1`. `autopay` checkbox + `lead_days` reminder.
    3. тЬЕ Attach the scan as `Media` on the rule via polymorphic `mediables` тАФ `inspector-open` with `mediaId` parameter pre-fills from `Media::ocr_extracted` (vendor, amount, dates, category hint written by the Tier 2 OCR job).
    4. тЬЕ Projection is materialized immediately by `saveBill` тЖТ `materializeInitialProjection`; lights up on Bills & Income + attention radar.
    5. ЁЯзн "Mark paid" action on the projection creates the matching Transaction with all the right fields pre-filled, and sets `projection.status=matched` / `matched_transaction_id` linking them.
    6. тЬЕ Direct entry from Mail inbox: mail rows with unlinked attachments expose "Create bill" / "Create transaction" buttons that open the Inspector pre-loaded with the media id.
  - **Non-bill rules** (maintenance, event, warranty) ignore the issue/due split тАФ for them `issued_on = due_on` and `due_offset_days = 0`. No behavior change.
  - **Closing the payment loop тАФ transaction тЖФ projection linkage**:
    - Schema has `recurring_projections.matched_transaction_id` / `matched_transfer_id` today but nothing writes them. Need the UI + auto-matcher to light it up.
    - **Mark paid** action on Bills & Income rows тАФ opens the Inspector Transaction form pre-filled from the projection (amount, counterparty from rule, category from rule, account from rule, reference from the ingested scan if any). On save: `projection.status='matched'`, `matched_transaction_id = newTxn.id`.
    - **Auto-match on Transaction create** тАФ when the Inspector saves a Transaction, scan for projections with `status in (projected, overdue)` matching same account + exact amount + `occurred_on` within ┬▒3d (optionally boosted by counterparty match). Exactly one hit тЖТ link automatically; multiple hits тЖТ show a "which bill did this pay?" selector before closing the drawer; zero hits тЖТ proceed as standalone.
    - **Attention radar** excludes `matched` projections entirely. Bills & Income renders them with a green "paid" chip + click-through to the matched Transaction in the Inspector.
    - **Bank-feed reconciliation** (per the Banking TODO) feeds into the same matcher: an incoming cleared bank row that matches a projection promotes it to `matched`, even for autopay rows that never had a manual transaction first.
    - **Audit**: `projection.matched_at` + `projection.unmatched_at` timestamps so the history is preserved if a user manually unlinks.
    - тЬЕ **Per-rule match tolerance** тАФ `recurring_rules.match_tolerance_days` (nullable small int; null тЖТ fall back to the app-wide default of 3). Utilities that vary by a whole week depending on the billing cycle can carry a wider tolerance without loosening the global default. Matcher reads the rule's value when comparing `projection.due_on` against `transaction.occurred_on`. Migration `2026_04_18_040000_add_match_tolerance_to_recurring_rules.php`.
    - ЁЯзн **Adaptive RRULE anchor (drift detection)** тАФ after N consecutive matches (say тЙе3) where `transaction.occurred_on = projection.due_on + delta` with a same-sign non-zero delta, nudge the rule's anchor so projections track reality without the user editing every rule. Two implementations, pick whichever lands first: either rewrite the rule's `BYMONTHDAY` (for small integer deltas on monthly rules), or introduce `recurring_rules.anchor_drift_days` (signed, applied on top of the RRULE-emitted date) for gentler, reversible nudges. Requires a populated match history to work тАФ `projection.matched_at` + `matched_transaction_id` are the inputs.
  - тЬЕ **Reconciliation workbench** тАФ `/reconcile` page with four orphan sections, count-tile header, and inline actions:
    - тЬЕ **Unmatched overdue projections** тЖТ "Mark paid" opens the Inspector transaction form (existing `inspector-mark-paid` path).
    - тЬЕ **Stale pending transactions** (>7d) тЖТ flip to cleared, open in inspector, or delete.
    - тЬЕ **Uncategorised cleared transactions** тЖТ open in inspector to pick a category.
    - тЬЕ **Half-done transfers** (status=pending >7d) тЖТ open in inspector to confirm both legs.
    - ЁЯзн Split-pane picker with side-by-side candidate comparison and keyboard nav is deferred тАФ the current single-column list with inline actions covers the common case; upgrade when reconciliation volume warrants it.
    - ЁЯзн Attention-radar badge ("N items need reconciliation") тАФ counts are surfaced on the workbench page itself; radar tile still to wire.

- ЁЯТб **Performance optimization pass (post feature-complete)** тАФ dedicated sweep once behavior is stable. Targets in priority order:
  - **Hot-path N+1 audits.** Profile `/reconcile`, `/transactions`, `/import/statements` preview, `/bills`, `/documents`, `/media` under realistic loads (тЙе 5k transactions, тЙе 500 contacts, тЙе 1k media). Any hit тЙе 100 queries on a normal render is a bug. Spot-fix with eager-load (`with(...)`) and bulk pre-loads тАФ the reconcile `reresolveUnreconciledByPattern` fix (3 queries regardless of N) is the reference pattern.
  - **Index audit.** Enable MariaDB slow-query log + `EXPLAIN`-review every query taking > 50 ms on the hot pages. Current schema has composite indexes on the obvious hot tuples (`(household_id, occurred_on)`, `(household_id, reconciled_at)`, `(household_id, category_id, occurred_on)`) but domain additions haven't always been re-audited. Likely gaps: `recurring_projections(status, due_on)`, `tags.taggable_type`, `contacts.match_patterns`-like searches if we ever move pattern matching to SQL.
  - **Livewire payload shrink.** `inspector.blade.php` (4k+ lines) carries dozens of public properties across every request; the wire-snapshot grows linearly. Fix options: split into per-type nested components, or drop derivable props (computed via `#[Computed]`) rather than persisting state. Target: < 30 KB snapshot on a typical drawer open.
  - **Computed caching across requests.** Several `#[Computed]` methods (`contactOptions`, `categoryPickerOptions`, `patternList`) walk the same table on every render. Wrap in `Cache::tags(...)` keyed on `household_id` + mtime, invalidate on `Contact`/`Category` save observers. Safe gain on a page that re-renders on every keystroke.
  - **OCR pipeline throughput.** Tesseract jobs serialize through a single queue worker today. Measure on a realistic backlog (50+ scans). Move to a dedicated queue (`ocr`) + bump workers when the 4090 laptop arrives.
  - **Frontend bundle.** Vite build size + initial paint on `/`, `/reconcile`, `/import/statements`. Alpine + Livewire + feature JS. Likely wins: lazy-load `searchable-select` Alpine factory only on pages that use it; drop unused Tailwind utilities via safelist tightening; treeshake the icon sets.
  - **Backup + snapshot job timings.** Nightly `backup:run` + monthly `snapshots:verify-restore` тАФ make sure neither drifts past its cron slot as the DB grows. Add telemetry on job duration.
  - **Livewire v4 morph.** Make sure big tables (transactions-index, reconcile) use `wire:key` correctly and don't re-mount children on every poll.

  Not a prerequisite for shipping features тАФ schedule alongside the code-reorg pass. Either can happen first, but the perf pass is more valuable once the UI surface area stabilises (fewer churned files тЖТ more representative profiles).

## Authentication & identity

Standard Laravel password auth ships in v1. Everything else is deferred.

- ЁЯзн **Social login (OAuth)** тАФ Laravel Socialite with Google, Apple, GitHub, Microsoft. "Sign in with Apple" is table stakes for iOS users. Sign-in is distinct from email ingestion OAuth (`integrations` table); keep them cleanly separated.

Ordered by ROI for the big-picture goal.

3. тЬЕ **Cross-domain tags** тАФ tag chips in drill-downs + tag hub (`/tags`, `/tags/{slug}`) walking 11 taggable models.
4. ЁЯПЧ **Reminders engine** тАФ multi-channel delivery (email, Slack, SMS, push, in-app, Telegram). Feeds attention radar.
5. ЁЯПЧ **OCR + full-text search on media** тАФ Tier 1 shipped: Tesseract via `App\Support\Ocr` + `App\Jobs\OcrMedia`, writes to `media.ocr_text`, surfaces in global search with snippet + deep-link. Tier 2 (local LLM) and Tier 3 (vision) still deferred:
    - **Tier 1 (no LLM)**: Tesseract OCR via `thiagoalessio/tesseract_ocr`, stored on `media.ocr_text` (new column) + Meilisearch/Typesense index. Makes every scanned receipt / bill / letter text-searchable through the existing `/` palette. Deterministic, offline, fast. Structured extraction relies on regex + per-vendor templates тАФ fragile, soтАж
    - **Tier 2 (+ local LLM)**: Tesseract extracts text, a local Ollama model (Llama/Mistral/Qwen) turns it into JSON (`amount, date, vendor, line_items, tax_lines, category_suggestion`), which pre-fills the Bill or Transaction form in the Inspector. Feeds the same MCP tool surface the AI assistant uses.
    - **Tier 3 (vision LLM)**: Skip Tesseract entirely for hard cases (handwritten, photographed, multi-column). Local multimodal model (LLaVA / Gemma-Vision via Ollama) parses the image directly.
    - **Explicit non-goal**: cloud vision APIs (Claude/GPT-4V/Gemini) as the default тАФ only opt-in per-document when the user declassifies.
6. ЁЯзн **Email ingestion beyond Postmark** тАФ Gmail API (OAuth) for existing-history scanning; IMAP/JMAP for Fastmail.
7. ЁЯзн **Bank CSV/OFX import** тАФ eliminate manual transaction entry. Plaid/SaltEdge as a later, heavier step.
    - ЁЯТб **Client-side batching for large statement drops** тАФ backfilling 7 ├Ч 12 = 84 PDFs hits PHP's `max_file_uploads=20` cap. Chunk the picked FileList client-side into batches of ~15, POST each via Livewire sequentially with a "batch 3 of 6" progress indicator, then compose the full parsed state once all batches land. Non-atomic by design: if batch 4 fails, batches 1тАУ3 stay imported and the UI offers a "retry remaining" affordance rather than rolling everything back.
8. ЁЯзн **Automations / rules engine** тАФ "when transaction matches merchant Y, auto-category Z"; "when contract expires in 30 days, create task"; event-driven side effects.
9. ЁЯзн **Dashboards (per-domain radars + one unified "state of affairs")** тАФ the primary UX. Drill-downs come off the dashboard.
10. ЁЯзн **Timeline view** ("what happened on this date last year") тАФ across all domains.
11. тЬЕ **Mobile PWA capture** тАФ `/m` shell + manifest + service worker + bottom tabs. Shipped surfaces: photo-inventory bulk (`/m/capture/inventory`), voice + text note (`/m/capture/note` with SpeechRecognition auto-restart), photo тЖТ Media with OCR queue (`/m/capture/photo`), Inbox tab. "Snap a shelf тЖТ suggested names" flow still ЁЯТб.
12. ЁЯзн **Sharing / partial visibility** тАФ read-only spouse access to selected domains.
13. ЁЯзн **Encryption at rest for media** + strong backup story (off-site, encrypted).
14. ЁЯзн **Weekly review prompt** тАФ surfaces orphan tasks, unreconciled transactions, stale contracts, expiring documents.
15. ЁЯТб **Natural-language interface via local LLM + MCP tools** тАФ expose Secretaire as a local MCP server so an on-device model (Ollama / llama.cpp / LM Studio) can do data management and tasking through structured tool calls. Nothing leaves the machine.
    - **Tool surface (per domain, CRUD + query)**: `transaction.create`, `transaction.find(filters)`, `task.create`, `task.update`, `contact.find`, `document.find_by_tag`, `account.balance_at`, `recurring.upcoming`, `note.capture`, etc. Typed JSON, scoped to the current household.
    - **Use cases**:
      - Capture: "spent $50 at Whole Foods yesterday, groceries" тЖТ `transaction.create`.
      - Query: "how much did I spend dining out this month" тЖТ `transaction.find` + sum.
      - Tasking: "remind me to change the HVAC filter in 3 weeks" тЖТ `task.create` + `reminder.create` (subject = Property).
      - Synthesis: "what's my net worth trend over the last 6 months" тЖТ `snapshot.find` + narrative summary.
      - Maintenance: "categorize last week's uncategorized transactions" тЖТ loops `transaction.find` + `transaction.update`.
    - **Guardrails**: every tool write respects `period_locks`, `household_id` scoping, and the download-gating rule. Every invocation lands in an `ai_tool_invocations` audit log (tool, arguments, outcome, latency, caller-client) so the CPA / user can see what the model touched.
    - **Clients**: Claude Desktop / LM Studio / Cursor pointing at the local MCP endpoint; later a first-party Livewire chat panel in Secretaire itself.
    - **Blocked on**: drill-down UX stable enough that tool contracts reflect the actual domain surface, and backups running (hallucinated deletes must be survivable).
16. ЁЯзн **Forecast / prognosis engine** тАФ project future income, spending, and balance from historical data + seasonality.
    - **Deterministic layer** тАФ `recurring_projections` already gives a known-future view for bills, income, transfers, maintenance. Sum forward across accounts тЖТ guaranteed-cashflow line.
    - **Probabilistic layer** тАФ learn from historical transactions to estimate non-recurring spending by category and month-of-year: moving averages per category, seasonality decomposition (December gift spike, summer travel, Q1 tax season), day-of-week patterns where relevant. Simple statistical methods (STL decomposition, exponential smoothing, or Prophet for richer cases) тАФ no ML needed for v1.
    - **Outputs** stored in `snapshots` (kind=`forecast`, payload = point estimate + confidence band, generated daily or on-demand):
      - End-of-month balance per account
      - 30/60/90-day cashflow prognosis (net in/out by week)
      - Year-end net-worth projection
      - Per-category spending forecast vs. typical
    - **Surfaces it drives**:
      - "If the next 30 days behave like the last 12 months, you'll end the month at $X" tile on the money radar.
      - Anomaly flags: "grocery spend is 40% above your seasonal baseline."
      - Runway estimates: "at current burn, liquid savings last N months if income stops."
      - Seasonal awareness: roadmap view showing expected expensive months ahead.
    - **Back-test column on `snapshots`** тАФ when a forecast date arrives, store actual vs. forecast so accuracy improves over time (and surfaces as "forecast confidence" on the dashboard).

- ЁЯзн **Reconciliation between manual entries and bank feeds** тАФ when a bank row arrives, try to match an existing manual row before inserting.

  - **Schema additions**: `transactions.external_id` (nullable unique-per-integration), `transactions.source` enum (`manual|bank_feed|import|inferred|reconciled`), `transactions.matched_at` timestamp, plus an Inspector "link / unlink" action for manual pairing when auto-match misses.
  - **Manual-entry duplicate guard**: on Inspector save, check for same-account / same-amount / ┬▒3d already `cleared` from a feed and warn before inserting.
  - **Deferred edge cases**: tip discrepancy (manual $50, bank $57), split transactions (match at parent), merchant-consolidated charges, refunds / chargebacks.

## Guardrails (schema conventions тАФ apply to every new domain)

- Canonical timeline column (`occurred_on` / `due_at` / `starts_at` / `effective_at` / `expires_on`) on anything that can appear in a radar.
- Money columns are always `amount` + `currency`; signed where direction matters.
- Status enums over free text for any field that will appear in a rollup dimension.
- Every tenant-scoped model uses `BelongsToHousehold` trait + `household_id` column.
- Polymorphic `taggables` + `mediables` тАФ every domain model is taggable and mediable unless there's an affirmative reason not to be.
- Datetimes stored in UTC; local conversion at the presentation layer only.
- **Every download link is gated** тАФ any route that returns binary or exported content (media, document, scan, statement PDF, export ZIP, backup archive, in-case-of pack) runs behind `auth` + `household` middleware, enforces per-entity ACL, and uses signed URLs with short TTLs for anything shared externally. No public file routes.
