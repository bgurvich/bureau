# Bureau — Roadmap

Personal-affairs management app. **Primary purpose: grasp the big picture of life state** — synthesis, dashboards, timelines. Detail capture is a means, not the end.

Stack: Laravel 13 · Livewire 4 · Vite 8 · MariaDB · stancl/tenancy (single-DB v1 → DB-per-tenant at SaaS).

Status legend: ✅ in v1 schema · 🏗 v1 wiring (post-schema) · 🧭 deferred, schema-reserved · 💡 brainstormed, no schema yet.

- ✅ recurring_projections (materialized upcoming occurrences → time radar)
- 🏗 **Bills as first-class surface (unified recurring + one-off)** — a "Bill" is a `recurring_rule` plus its next `recurring_projection`, no new table. Schema + autopay UX shipped; paper-bill intake + auto-matcher + reconciliation workbench still pending.
  - **DTSTART = issue date** (invoice date, when the bill was created by the counterparty), **not** the due date. The two are distinct and both matter (issue = accrual, due = cash-flow).
  - Recurring bill: RRULE anchored on the typical issue day (`FREQ=MONTHLY;BYMONTHDAY=1` — landlord issues on the 1st). Each occurrence has a due date computed as `issued_on + due_offset_days`.
  - One-off bill (medical, repair, surprise invoice): RRULE `FREQ=DAILY;COUNT=1`, DTSTART = the actual issue date on the paper. Expander emits exactly one projection on the issue date; its `due_on` is issue + `due_offset_days`. Rule effectively inert after its single projection matches a transaction.
  - **Schema additions** (✅ shipped via `2026_04_17_116000_add_bill_fields_to_recurring_tables.php`):
    - ✅ `recurring_rules.due_offset_days` (unsigned small int, default 0) — days between issue and due for occurrences generated by this rule.
    - ✅ `recurring_rules.autopay` (bool, default false) — bill is on the counterparty's auto-charge; no manual payment required. Propagates to projections.
    - ✅ `recurring_projections.issued_on` (nullable date) — the expanded DTSTART date for that occurrence (= RRULE anchor). `due_on` stays as the payable date.
    - ✅ `recurring_projections.autopay` (bool, default false) — denormalized from the rule at projection time so later edits to the rule don't rewrite history.
    - ✅ The `recurring:project` command fills `issued_on` with the expander's date, `due_on = issued_on + rule.due_offset_days`, and copies `autopay` from the rule.
  - **Autopay behavior** (✅ shipped):
    - ✅ Attention radar skips autopay projections until they're more than 7 days past `due_on` without a matched transaction — at which point something failed (expired card, declined charge) and the user genuinely needs to look.
    - ✅ Bills & Income page still renders autopay projections but in a muted "informational" row style with an `AUTO` chip, so the user sees upcoming burn without being nudged.
    - 🧭 Money radar's "next 30 days out" cumulative burn includes autopay projections (they still hit the account).
    - 🧭 When bank feeds + reconciliation land, autopay projections auto-promote to `matched` on any incoming bank row that satisfies the match rule.
  - ✅ **Paper-bill intake workflow**:
    1. ✅ Bill arrives (mail, email, or verbal). User hits `⌘.` → **Bill** in the Inspector type picker, or `B` keyboard shortcut on `/bills`.
    2. ✅ Form fields shipped: `counterparty` (searchable-select with allow-create), `amount`, `issued_on` (defaults today), `due_on` (required; inspector computes `due_offset_days` from the two dates), `category`, `account`, `is_recurring` toggle → monthly/weekly/yearly frequency + optional stop-after; off = `COUNT=1`. `autopay` checkbox + `lead_days` reminder.
    3. ✅ Attach the scan as `Media` on the rule via polymorphic `mediables` — `inspector-open` with `mediaId` parameter pre-fills from `Media::ocr_extracted` (vendor, amount, dates, category hint written by the Tier 2 OCR job).
    4. ✅ Projection is materialized immediately by `saveBill` → `materializeInitialProjection`; lights up on Bills & Income + attention radar.
    5. 🧭 "Mark paid" action on the projection creates the matching Transaction with all the right fields pre-filled, and sets `projection.status=matched` / `matched_transaction_id` linking them.
    6. ✅ Direct entry from Mail inbox: mail rows with unlinked attachments expose "Create bill" / "Create transaction" buttons that open the Inspector pre-loaded with the media id.
  - **Non-bill rules** (maintenance, event, warranty) ignore the issue/due split — for them `issued_on = due_on` and `due_offset_days = 0`. No behavior change.
  - **Closing the payment loop — transaction ↔ projection linkage**:
    - Schema has `recurring_projections.matched_transaction_id` / `matched_transfer_id` today but nothing writes them. Need the UI + auto-matcher to light it up.
    - **Mark paid** action on Bills & Income rows — opens the Inspector Transaction form pre-filled from the projection (amount, counterparty from rule, category from rule, account from rule, reference from the ingested scan if any). On save: `projection.status='matched'`, `matched_transaction_id = newTxn.id`.
    - **Auto-match on Transaction create** — when the Inspector saves a Transaction, scan for projections with `status in (projected, overdue)` matching same account + exact amount + `occurred_on` within ±3d (optionally boosted by counterparty match). Exactly one hit → link automatically; multiple hits → show a "which bill did this pay?" selector before closing the drawer; zero hits → proceed as standalone.
    - **Attention radar** excludes `matched` projections entirely. Bills & Income renders them with a green "paid" chip + click-through to the matched Transaction in the Inspector.
    - **Bank-feed reconciliation** (per the Banking TODO) feeds into the same matcher: an incoming cleared bank row that matches a projection promotes it to `matched`, even for autopay rows that never had a manual transaction first.
    - **Audit**: `projection.matched_at` + `projection.unmatched_at` timestamps so the history is preserved if a user manually unlinks.
    - ✅ **Per-rule match tolerance** — `recurring_rules.match_tolerance_days` (nullable small int; null → fall back to the app-wide default of 3). Utilities that vary by a whole week depending on the billing cycle can carry a wider tolerance without loosening the global default. Matcher reads the rule's value when comparing `projection.due_on` against `transaction.occurred_on`. Migration `2026_04_18_040000_add_match_tolerance_to_recurring_rules.php`.
    - 🧭 **Adaptive RRULE anchor (drift detection)** — after N consecutive matches (say ≥3) where `transaction.occurred_on = projection.due_on + delta` with a same-sign non-zero delta, nudge the rule's anchor so projections track reality without the user editing every rule. Two implementations, pick whichever lands first: either rewrite the rule's `BYMONTHDAY` (for small integer deltas on monthly rules), or introduce `recurring_rules.anchor_drift_days` (signed, applied on top of the RRULE-emitted date) for gentler, reversible nudges. Requires a populated match history to work — `projection.matched_at` + `matched_transaction_id` are the inputs.
  - ✅ **Reconciliation workbench** — `/reconcile` page with four orphan sections, count-tile header, and inline actions:
    - ✅ **Unmatched overdue projections** → "Mark paid" opens the Inspector transaction form (existing `inspector-mark-paid` path).
    - ✅ **Stale pending transactions** (>7d) → flip to cleared, open in inspector, or delete.
    - ✅ **Uncategorised cleared transactions** → open in inspector to pick a category.
    - ✅ **Half-done transfers** (status=pending >7d) → open in inspector to confirm both legs.
    - 🧭 Split-pane picker with side-by-side candidate comparison and keyboard nav is deferred — the current single-column list with inline actions covers the common case; upgrade when reconciliation volume warrants it.
    - 🧭 Attention-radar badge ("N items need reconciliation") — counts are surfaced on the workbench page itself; radar tile still to wire.

## Authentication & identity

Standard Laravel password auth ships in v1. Everything else is deferred.

- 🧭 **Social login (OAuth)** — Laravel Socialite with Google, Apple, GitHub, Microsoft. "Sign in with Apple" is table stakes for iOS users. Sign-in is distinct from email ingestion OAuth (`integrations` table); keep them cleanly separated.
- 🧭 **Magic link login** — passwordless email links. Laravel has it via `Auth::loginUsingId()` + signed URLs; `laravel/fortify` bundles a ready flow. Useful for a family member who rarely logs in.
- 🧭 **2FA** — TOTP (Google Authenticator / 1Password) via `laravel/fortify`. Recovery codes. Enforced per-household-role (owners required, members optional).
- 🧭 **Passkeys (WebAuthn)** — phase-in after 2FA. `laravel/fortify` + a WebAuthn driver. Biometric unlock on iOS/Android is the end-state for a personal app with sensitive data.

Ordered by ROI for the big-picture goal.

4. 🏗 **Reminders engine** — multi-channel delivery (email, Slack, SMS, push, in-app, Telegram). Feeds attention radar.

    - **Tier 1 (no LLM)**: Tesseract OCR via `thiagoalessio/tesseract_ocr`, stored on `media.ocr_text` (new column) + Meilisearch/Typesense index. Makes every scanned receipt / bill / letter text-searchable through the existing `/` palette. Deterministic, offline, fast. Structured extraction relies on regex + per-vendor templates — fragile, so…
    - **Tier 2 (+ local LLM)**: Tesseract extracts text, a local Ollama model (Llama/Mistral/Qwen) turns it into JSON (`amount, date, vendor, line_items, tax_lines, category_suggestion`), which pre-fills the Bill or Transaction form in the Inspector. Feeds the same MCP tool surface the AI assistant uses.
    - **Tier 3 (vision LLM)**: Skip Tesseract entirely for hard cases (handwritten, photographed, multi-column). Local multimodal model (LLaVA / Gemma-Vision via Ollama) parses the image directly.
    - **Explicit non-goal**: cloud vision APIs (Claude/GPT-4V/Gemini) as the default — only opt-in per-document when the user declassifies.
6. 🧭 **Email ingestion beyond Postmark** — Gmail API (OAuth) for existing-history scanning; IMAP/JMAP for Fastmail.
7. 🧭 **Bank CSV/OFX import** — eliminate manual transaction entry. Plaid/SaltEdge as a later, heavier step.
    - 💡 **Client-side batching for large statement drops** — backfilling 7 × 12 = 84 PDFs hits PHP's `max_file_uploads=20` cap. Chunk the picked FileList client-side into batches of ~15, POST each via Livewire sequentially with a "batch 3 of 6" progress indicator, then compose the full parsed state once all batches land. Non-atomic by design: if batch 4 fails, batches 1–3 stay imported and the UI offers a "retry remaining" affordance rather than rolling everything back.
8. 🧭 **Automations / rules engine** — "when transaction matches merchant Y, auto-category Z"; "when contract expires in 30 days, create task"; event-driven side effects.
9. 🧭 **Dashboards (per-domain radars + one unified "state of affairs")** — the primary UX. Drill-downs come off the dashboard.
10. 🧭 **Timeline view** ("what happened on this date last year") — across all domains.
11. ✅ **Mobile PWA capture** — `/m` shell + manifest + service worker + bottom tabs. Shipped surfaces: photo-inventory bulk (`/m/capture/inventory`), voice + text note (`/m/capture/note` with SpeechRecognition auto-restart), photo → Media with OCR queue (`/m/capture/photo`), Inbox tab. "Snap a shelf → suggested names" flow still 💡.
12. 🧭 **Sharing / partial visibility** — read-only spouse access to selected domains.
13. 🧭 **Encryption at rest for media** + strong backup story (off-site, encrypted).
14. 🧭 **Weekly review prompt** — surfaces orphan tasks, unreconciled transactions, stale contracts, expiring documents.
15. 💡 **Natural-language interface via local LLM + MCP tools** — expose Bureau as a local MCP server so an on-device model (Ollama / llama.cpp / LM Studio) can do data management and tasking through structured tool calls. Nothing leaves the machine.
    - **Tool surface (per domain, CRUD + query)**: `transaction.create`, `transaction.find(filters)`, `task.create`, `task.update`, `contact.find`, `document.find_by_tag`, `account.balance_at`, `recurring.upcoming`, `note.capture`, etc. Typed JSON, scoped to the current household.
    - **Use cases**:
      - Capture: "spent $50 at Whole Foods yesterday, groceries" → `transaction.create`.
      - Query: "how much did I spend dining out this month" → `transaction.find` + sum.
      - Tasking: "remind me to change the HVAC filter in 3 weeks" → `task.create` + `reminder.create` (subject = Property).
      - Synthesis: "what's my net worth trend over the last 6 months" → `snapshot.find` + narrative summary.
      - Maintenance: "categorize last week's uncategorized transactions" → loops `transaction.find` + `transaction.update`.
    - **Guardrails**: every tool write respects `period_locks`, `household_id` scoping, and the download-gating rule. Every invocation lands in an `ai_tool_invocations` audit log (tool, arguments, outcome, latency, caller-client) so the CPA / user can see what the model touched.
    - **Clients**: Claude Desktop / LM Studio / Cursor pointing at the local MCP endpoint; later a first-party Livewire chat panel in Bureau itself.
    - **Blocked on**: drill-down UX stable enough that tool contracts reflect the actual domain surface, and backups running (hallucinated deletes must be survivable).
16. 🧭 **Forecast / prognosis engine** — project future income, spending, and balance from historical data + seasonality.
    - **Deterministic layer** — `recurring_projections` already gives a known-future view for bills, income, transfers, maintenance. Sum forward across accounts → guaranteed-cashflow line.
    - **Probabilistic layer** — learn from historical transactions to estimate non-recurring spending by category and month-of-year: moving averages per category, seasonality decomposition (December gift spike, summer travel, Q1 tax season), day-of-week patterns where relevant. Simple statistical methods (STL decomposition, exponential smoothing, or Prophet for richer cases) — no ML needed for v1.
    - **Outputs** stored in `snapshots` (kind=`forecast`, payload = point estimate + confidence band, generated daily or on-demand):
      - End-of-month balance per account
      - 30/60/90-day cashflow prognosis (net in/out by week)
      - Year-end net-worth projection
      - Per-category spending forecast vs. typical
    - **Surfaces it drives**:
      - "If the next 30 days behave like the last 12 months, you'll end the month at $X" tile on the money radar.
      - Anomaly flags: "grocery spend is 40% above your seasonal baseline."
      - Runway estimates: "at current burn, liquid savings last N months if income stops."
      - Seasonal awareness: roadmap view showing expected expensive months ahead.
    - **Back-test column on `snapshots`** — when a forecast date arrives, store actual vs. forecast so accuracy improves over time (and surfaces as "forecast confidence" on the dashboard).

### Banking
- 🧭 CSV / OFX import (first wave — manual per statement).
- 🧭 Plaid (US) or GoCardless Bank Account Data (EU) — automated feeds.
- 🧭 SaltEdge — alternative with broad geographic coverage.
- 🧭 **Reconciliation between manual entries and bank feeds** — when a bank row arrives, try to match an existing manual row before inserting.
  - **Match rule**: same `account_id` + exact `amount` + `occurred_on` within ±3 days (tolerates posting delay).
  - **On match**: the manual row's `status` flips `pending` → `cleared`, the bank's canonical date/amount overwrite the row, and the bank's `external_id` is stamped on it. User-entered category / counterparty / reference / media are preserved.
  - **No match**: insert the bank row with `status='cleared'`, `source='bank_feed'`, no category — it shows up on the attention radar as "uncategorized" for the user to label.
  - **Schema additions**: `transactions.external_id` (nullable unique-per-integration), `transactions.source` enum (`manual|bank_feed|import|inferred|reconciled`), `transactions.matched_at` timestamp, plus an Inspector "link / unlink" action for manual pairing when auto-match misses.
  - **Manual-entry duplicate guard**: on Inspector save, check for same-account / same-amount / ±3d already `cleared` from a feed and warn before inserting.
  - **Deferred edge cases**: tip discrepancy (manual $50, bank $57), split transactions (match at parent), merchant-consolidated charges, refunds / chargebacks.

## Guardrails (schema conventions — apply to every new domain)

- Canonical timeline column (`occurred_on` / `due_at` / `starts_at` / `effective_at` / `expires_on`) on anything that can appear in a radar.
- Money columns are always `amount` + `currency`; signed where direction matters.
- Status enums over free text for any field that will appear in a rollup dimension.
- Every tenant-scoped model uses `BelongsToHousehold` trait + `household_id` column.
- Polymorphic `taggables` + `mediables` — every domain model is taggable and mediable unless there's an affirmative reason not to be.
- Datetimes stored in UTC; local conversion at the presentation layer only.
- **Every download link is gated** — any route that returns binary or exported content (media, document, scan, statement PDF, export ZIP, backup archive, in-case-of pack) runs behind `auth` + `household` middleware, enforces per-entity ACL, and uses signed URLs with short TTLs for anything shared externally. No public file routes.
